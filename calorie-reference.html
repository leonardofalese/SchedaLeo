1 · Come funziona il parser degli alimenti

Struttura di una voce alimento

Ogni stringa nella scheda ha il formato [quantità][unità] [nome alimento]. Il parser la scompone in 4 passi:

PASSO 1 — Kcal esplicite? Se la stringa contiene "kcal" (es. "500 kcal") → usa direttamente, stop. PASSO 2 — Estrai quantità + unità Regex: ^(\d+(?:[.,]\d+)?)\s*(g|kg|ml|l|pz|fette|scatolette|buste|pacco|pacchetto|porzione)? "150g petto di pollo" → qty=150 unit="g" food="petto di pollo" "2 scatolette tonno" → qty=2 unit="scatolette" food="tonno" "1 frutto" → qty=1 unit="pz" food="frutto" "3 fette pane" → qty=3 unit="fette" food="pane" Se unità non trovata → default "g" Se nessun numero → return 0 PASSO 3 — Converti unità → grammi (vedi tabella sezione 2) PASSO 4 — Cerca kcal/100g nel DB, calcola kcal = round( grammi × kcal_per_100g / 100 ) (vedi sezione 3 per il lookup)

3 · Lookup nel database (come trova l'alimento)

Ordine di ricerca — 3 tentativi

foodRaw = stringa dopo la quantità, con tutti gli aggettivi es. "250g macinato manzo magro" → foodRaw = "macinato manzo magro" foodClean = foodRaw senza stopword connettive (con/al/di/della/del/no/senza…) es. "pasta al pomodoro" → foodClean = "pasta pomodoro" Tentativo 1: cerca foodRaw nel DB Tentativo 2: cerca foodClean nel DB Tentativo 3: cerca l'intera stringa originale (str) Fallback: 150 kcal/100g se nessun match trovato

Priorità: chiave più lunga vince

KCAL_DB_SORTED = Object.entries(KCAL_DB).sort((a,b) => b[0].length - a[0].length) Esempio con "pasta integrale": Chiave "pasta integrale" (16 char) → 340 kcal ✓ cercata PRIMA Chiave "pasta" (5 char) → 350 kcal sarebbe sbagliata se trovata prima Esempio con "macinato manzo magro": "macinato manzo magro" (20 char) → 178 kcal ✓ "macinato manzo" (14 char) → 248 kcal "macinato" (8 char) → 248 kcal Differenza: 180 kcal su 250g — il match specifico è fondamentale
Stopwords rimosse per foodClean: con · al · alla · allo · agli · alle · di · del · della · dello · in · no · senza · e
NON rimosse: aggettivi come "magro", "integrale", "affumicato" — servono per matchare voci specifiche del DB.

4 · Esempi di parsing tracciati

"140g pasta integrale"
Regex → qty=140, unit=g, foodRaw="pasta integrale"
Lookup: "pasta integrale" (16ch) trovata → 340 kcal/100g
(se cercasse "pasta" prima → 350 — sbagliato)
= 476 kcal
"2 scatolette tonno no olio"
Regex → qty=2, unit=scatolette, foodRaw="tonno no olio"
Scatoletta tonno → 80g/cad → 2 × 80 = 160g
Lookup: "tonno in scatoletta" (19ch) vince → 103 kcal/100g
= 165 kcal
"250g macinato manzo magro"
Regex → qty=250g, foodRaw="macinato manzo magro"
Lookup tentativo 1 su foodRaw: "macinato manzo magro" (20ch) → 178 kcal/100g ✓
⚠ Se "magro" fosse una stopword, troverebbe "macinato manzo" (248) → errore di +175 kcal su 250g
= 445 kcal
"1 frutto"
Regex → qty=1, unit=pz, foodRaw="frutto"
PZ_WEIGHTS["frutto"] = 150g
Lookup: "frutto" → 55 kcal/100g
= 83 kcal
"3 fette pane in cassetta"
Regex → qty=3, unit=fette, foodRaw="pane in cassetta"
Non è biscotto → 35g/fetta → 3 × 35 = 105g
Lookup: "pane in cassetta" (16ch) → 268 kcal/100g
= 281 kcal
"1 pacchetto galbusera protein"
Regex → qty=1, unit=pacchetto, foodRaw="galbusera protein"
PACCO_WEIGHTS["galbusera protein"] = 32g (peso reale confezione)
Lookup: "galbusera protein" (17ch) → 362 kcal/100g
= 116 kcal
"10g olio"
Regex → qty=10g, foodRaw="olio"
Lookup: "olio evo" (8ch) cercato prima — ma "olio" da solo → match "olio" → 884 kcal/100g
⚠ L'olio è 8.84 kcal/g — 10g sembrano pochi ma fanno 88 kcal
= 88 kcal
"500 kcal"
Regex kcal esplicite → 500 estratte direttamente, nessun altro calcolo
= 500 kcal

6 · Calcolo BMR e TDEE base

BMR · Mifflin-St Jeor

Base = 10 × peso(kg) + 6.25 × altezza(cm) − 5 × età(anni) Uomo: BMR = Base + 5 Donna: BMR = Base − 161

Rappresenta il fabbisogno calorico a completo riposo (solo funzioni vitali). È la base su cui si applica il moltiplicatore di attività.

TDEE = BMR × moltiplicatore

TDEE = BMR × calcActivityMultiplier() Comprende: • attività base (frequenza allenamenti) • bonus volume palestra Bilancio/die = kcal_scheda − TDEE > 0 → surplus (accumulo) < 0 → deficit (dimagrimento)

kcal_scheda = media giornaliera dei 7 giorni della scheda (solo giorni con kcal > 0).

7 · Moltiplicatore attività palestra

calcActivityMultiplier(trainingDays, totalVol)

Step 1 — Moltiplicatore base per giorni di allenamento/settimana: 0 giorni → ×1.20 (sedentario) 1–2 gg → ×1.375 (attività leggera) 3–4 gg → ×1.55 (moderata) 5–6 gg → ×1.725 (attiva) 7 gg → ×1.90 (molto attiva) Step 2 — Bonus volume tonnellaggio (serie × rip × kg / settimana): volBonus = min( (totalVol / 10.000) × 0.015, 0.15 ) Esempi: 5.000 kg/sett → bonus +0.0075 10.000 kg/sett → bonus +0.015 50.000 kg/sett → bonus +0.075 (max +0.15 a 100.000 kg) Moltiplicatore finale = min(base + volBonus, 1.90) Esempio: 4 gg, vol=12.000 → ×1.55 + 0.018 = ×1.568
trainingDays = giorni con ≥1 esercizio e nome ≠ "riposo"
totalVol = somma su tutta la settimana di serie × ripetizioni × kg per ogni esercizio che ha tutti e 3 i campi compilati

8 · Proiezione obiettivo

Prerequisiti: profilo compilato (peso, altezza, età, sesso, obiettivo, peso obiettivo) + bilancio calorico calcolabile (scheda con dati)

Dimagrire — formula calorica

Principio: 1 kg grasso ≈ 7700 kcal deficit/die = |kcal_scheda − TDEE| (deve essere negativo) giorni = (kg_da_perdere × 7700) / deficit settimane = round(giorni / 7) → più allenamento → TDEE alto → TDEE alto → deficit grande → deficit grande → stima più corta ✓

Composizione stimata (fatRatio)

0 gg/sett → fatRatio = 0.62 1 gg/sett → fatRatio = 0.73 2–3 gg/sett→ fatRatio = 0.80 4+ gg/sett → fatRatio = 0.86 vol≥10.000 → fatRatio += 0.03 (max 0.92) estFatLoss = kgDiff × fatRatio estLeanLoss = kgDiff − estFatLoss (più ti alleni → meno massa magra perdi)

Massa — formula fisiologica

Principio: la crescita muscolare ha un limite ormonale/biologico, non solo calorico kg/mese base: 0 gg/sett → 0.25 kg/mese 1–2 gg/sett→ 0.40 kg/mese 3 gg/sett → 0.60 kg/mese 4 gg/sett → 0.75 kg/mese 5+ gg/sett → 0.90 kg/mese Bonus volume: totalVol ≥ 5.000 kg/sett → +0.08 (cap 1.20) totalVol ≥ 15.000 kg/sett → +0.10 (cap 1.50) Vincolo calorico (riduce il tasso): balance < 0 → ×0.30 (deficit: quasi impossibile) balance < 150 → ×0.55 (surplus insufficiente) balance < 300 → ×0.80 (borderline) balance ≥ 300 → tasso pieno ✓ mesi_stimati = kgDiff / kgPerMese

Composizione stimata (muscleRatio)

2–3 gg/sett → muscleRatio = 0.55 4 gg/sett → muscleRatio = 0.65 5+ gg/sett → muscleRatio = 0.72 vol≥10.000 → +0.05 (max 0.80) vol≥20.000 → +0.05 (max 0.80) estMuscle = kgDiff × muscleRatio estFat = kgDiff − estMuscle

Grafici proiezione

Massa — curva lineare settimanale kgPerWeek = kgPerMonth / 4.3 pts[i] = min(peso + kgPerWeek × i, pesoObiettivo) weeksMax = ceil(kgDiff / kgPerWeek) + 1 (cap 120 settimane) Dimagrire — curva lineare dal bilancio calorico pts[i] = peso + (balance / 7700) × 7 × i (balance è negativo → curva discendente) weeksMax = round((kgDiff × 7700) / (|balance| × 7)) + 1

Le label sull'asse X mostrano "Oggi", poi mesi (ogni 4.3 settimane), poi "Traguardo" all'ultima settimana.

9 · Tracker — come funziona

Struttura dati tracker

state.meals = { "d0_colazione": true, "d1_pranzo": false, … } → booleano per ogni pasto di ogni giorno state.gymLog = { "gym_d0_0": true, "gym_d2_3": false, … } → booleano per ogni esercizio di ogni giorno Chiave pasto: "d{dayIndex}_{meal_key}" es. "d0_colazione" Chiave gym: "gym_d{dayIndex}_{exIndex}" es. "gym_d1_2"

Grafici alimentazione

Bar chart settimanale — kcal totali dei 7 giorni della scheda (non di quelli completati — sono le kcal pianificate)
Doughnut pasti — media kcal per tipo di pasto su tutta la settimana (colazione/pranzo/spuntini/cena)

Grafici palestra

Completamento stacked — esercizi completati vs rimanenti per ogni giorno
Volume per giorno — serie × rip × kg per ogni giorno (solo se tutti i campi sono compilati)

Statistiche riepilogative

Alimentazione: BMR = Mifflin-St Jeor con profilo utente TDEE = BMR × actMult (include palestra) Scheda = media kcal giorni attivi della scheda Bilancio= Scheda − TDEE (kcal/die) Palestra: Sessioni/sett = giorni con esercizi (nome ≠ riposo) Esercizi/sett = totale esercizi su 7 giorni Serie totali = somma di tutte le serie Volume sett = somma serie × rip × kg (mostrato se > 0) Completato % = esercizi spuntati / totale esercizi

10 · Stato miglioramenti

Tutti i miglioramenti della roadmap originale sono stati implementati. Di seguito lo stato attuale e i successivi possibili sviluppi.
✓ Macronutrienti (P / C / G) Implementato
MACRO_DB con ~120 voci · MACRO_DB_SORTED per match più-specifico-prima · calcMacrosFromFood() restituisce {kcal,p,c,g} per qualsiasi stringa alimento.
Tracker mostra distribuzione settimanale P/C/G con barra percentuale e medie giornaliere. Pillole macro visibili anche nella home accanto al progresso kcal.
✓ Peso corporeo progressivo e andamento reale Implementato
state.weightLog: [{date, kg}] · input rapido peso (card compatta in home) · grafico peso ultime 30 rilevazioni nel tracker.
logWeight() aggiorna anche profileData.peso (usato per BMR/proiezione) al salvataggio.
✓ Progressione carichi palestra Implementato
state.gymHistory: {'YYYY-MM-DD': {'Nome esercizio': {kg, rip, serie}}} · bottone "Registra" nel pannello allenamento giornaliero · sezione "Progressione carichi" nel tracker con grafico Canvas per ogni esercizio (ultime 10 sessioni).
getExerciseHistory(nameLower) restituisce array cronologico per charting.
✓ Calibrazione TDEE su dati reali Implementato
calcCalibratedTDEE(): richiede ≥7 giorni di weightLog. Formula:
TDEE_reale = avgKcal − (ΔkgTotale × 7700 / giorni)
Se disponibile, sovrascrive il TDEE teorico Mifflin nel tracker con badge "✓ calibrato".
✓ DB sinonimi e alias Implementato
ALIASES: ~45 voci che mappano varianti di scrittura alla chiave canonica del DB (es. 'wurst'→'wurstel', 'carne macinata'→'macinato manzo', 'olio extravergine'→'olio evo', 'skyr'→'yogurt greco 0%').
_aliasNormalize(str) applicata come passo intermedio nel lookup — foodRaw → foodClean → foodAliased → str → 150 kcal fallback.

Prossimi sviluppi potenziali

Ricette personalizzate salvabili Alta priorità
Permettere di salvare un insieme di alimenti come "ricetta" con nome (es. "Pasta tonno zucchine") e inserirla nella scheda come voce singola.
Cosa serve: state.recipes: {nome: [food1, food2, ...]} · UI per creare/modificare ricette · lookup che espande la voce prima del calcolo kcal/macro.
Impatto: meno reinserimenti manuali per chi mangia spesso le stesse combinazioni.
Variabilità kcal — cicli alta/bassa Media priorità
Giorni di allenamento e riposo spesso richiedono kcal diverse (es. +200 kcal nei giorni di palestra). Attualmente il target è fisso.
Cosa serve: campo "kcal bonus" per tipo di giornata (allenamento / riposo / libero) nel profilo · calcDayTarget(dayIndex) che distingue la tipologia.
Peso sgocciolato per prodotti in scatola Bassa priorità
"1 scatoletta tonno" = 80g sgocciolato (hardcoded). Prodotti come legumi in scatola e polpa di pomodoro hanno pesi diversi.
Cosa serve: estendere SCATOLETTA_WEIGHTS con più prodotti noti (ceci ≈140g, lenticchie ≈120g, fagioli ≈130g).
Scan barcode tramite fotocamera Bassa priorità
Leggere EAN del prodotto e cercarne i valori nutrizionali via API Open Food Facts. Azzererebbe i fallback a 150 kcal per prodotti confezionati non nel DB locale.
Cosa serve: QuaggaJS o ZXing per lettura barcode · chiamata API OpenFoodFacts · cache locale dei prodotti scansionati.
Come verificare un calcolo sospetto: nella scheda, scrivi l'alimento e controlla le kcal calcolate nel pannello di modifica inline. Se ti sembra errato, confronta con la tabella sopra e verifica che la stringa faccia match con la chiave giusta. In alternativa, aggiungi una voce a ALIASES in js/config.js per normalizzare la variante di scrittura.
